# PDU Data Automation v0.2.15

Corrective pilot release: backend-owned CSV readiness and timers, strict System Burn-In sequencing, STEP42 Changeover Teams card, dynamic floor/admin identities, and clearer Advanced Settings.

## Changes

- **fix(automation): monotonic backend-owned CSV readiness**
  - Scan facts expose `process_ready`, wait phase, absolute phase deadline, and pending duration.
  - Direct and batch processing re-check readiness before any workbook write.
  - Not-ready results stay retryable waits; they are not persisted as processor outcomes.
  - React derives remaining time from backend anchors; rescans no longer restart a locked-CSV countdown.

- **fix(automation): strict System Burn-In STEP71 / STEP72**
  - STEP71 starts the two-hour soak; matching readable STEP72 must age one minute before report processing.
  - A STEP72-only unprocessed folder is not processed; historical passes remain passed.
  - Phase copy distinguishes soak, waiting for STEP72 capture, and unlock waits.

- **feat(notifications): one-time STEP42 Changeover card**
  - After a committed pass for `208v-breaker-8-20% Load` (STEP41), posts a Changeover Adaptive Card (default on).
  - Durable per-unit receipt; excluded from `shift_log.json` (Problem/Complete only).
  - Soft-fail only: notification issues never block automation or STEP43.

- **feat(settings): dynamic floor and admin identities**
  - Add Floor Station or Admin Identity; rename display names; stable backend-generated IDs.
  - First dynamic create upgrades `floor_settings.json` to schema 2.
  - Admin identities can be this PC’s local identity and appear on Test Ping / runtime cards, but cannot be Main poster or included end-of-shift stations.
  - Delete and role mutation are out of scope for this release.

- **feat(settings): Advanced page split + unified identity control**
  - Advanced menu: **Station & Identities**, **Teams & Notifications**, Change password.
  - Independent `identity` and `teams` save scopes so stale pages cannot overwrite each other.
  - Single searchable identity control for select, add, and explicit rename.
  - Identity-scope save can seed a missing `floor_settings.json` when the shared folder exists; ordinary loads still do not auto-reseed.

## Deploy notes for station PCs

1. Install `PDU Data Automation_0.2.15_x64-setup.exe` (S-drive root or GitHub), or use the in-app updater from **v0.2.10+** once this release is published with `latest.json` + `.sig`.
2. **Upgrade every PC that shares the same `.PDU_Notifications` folder before anyone creates a new identity.** v0.2.14 rejects schema 2 rather than corrupting it, but must not keep editing settings after schema 2 exists.
3. On each station: Advanced → Browse the same synced `.PDU_Notifications` folder if not already connected; Connect with **Existing floor password** when needed.
4. Confirm This PC identity; use **Station & Identities** to add/rename only after all peers are on v0.2.15.
5. Optional: Advanced → Teams → confirm Changeover toggle (default on) and send a Test ping.

## Not in this release

- Automatic scheduled end-of-shift posts at shift end times.
- Stuck / idle automation.
- Identity delete or role change after create.
- Full migration of system/breaker/burn-in cell maps into layout JSON (transformers remain the data-driven path).

## Validation

- Run full local validation before packaging: frontend tests/lint/build, Rust checks/tests, report-layout schema, version consistency.
- Version consistency `0.2.15` across `package.json` / `Cargo.toml` / `tauri.conf.json`.
- Operator smoke after install: locked CSV countdown does not restart; System Burn-In waits for STEP72; Changeover after STEP41 pass; identity add only after floor-wide upgrade.

## Compatibility

- Existing schema-1 floors remain readable; schema stays 1 until the first dynamic identity is created.
- Existing Problem/Complete shift log schema is unchanged.
- Updater signing key remains the v0.2.10 replacement key; pre-0.2.10 installs still need a one-time manual installer.
