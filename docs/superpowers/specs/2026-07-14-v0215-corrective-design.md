# v0.2.15 Corrective Automation, Changeover, and Dynamic Identity Design

**Status:** Approved for implementation on 2026-07-14
**Current released baseline:** v0.2.14 (`10d1a4a`)
**Implementation branch:** `fix/v0.2.15-corrective`
**Related implementation plan:** `docs/superpowers/plans/2026-07-14-v0215-corrective.md`

## Purpose

This corrective slice fixes four production-observed gaps without changing the familiar PDU operator workflow:

1. A detected CSV must keep one monotonic countdown even while the file is locked or still being written; a three-second rescan must never restart that countdown.
2. System Burn-In must require STEP71's two-hour soak and then a matching, readable STEP72 capture with its own one-minute wait before report processing.
3. A one-time Teams Changeover card must be sent after the committed pass for `208v-breaker-8-20% Load` (STEP41), representing the manual STEP42 shutdown/tap change before STEP43.
4. Notification identities must be a dynamic shared catalog. Advanced Settings must use editable/searchable comboboxes to rename existing identities or add Floor Station and Admin Identity entries while preserving stable IDs.

The corrective work is intentionally not a release operation. The repository remains at application version `0.2.14` during implementation. A version bump, installer build, signing, tag, GitHub Release, S-drive staging, or deployment requires separate authorization after verification.

## Locked product decisions

- The readiness clock is backend-owned and timestamp-based. React displays time from backend anchors and a one-second UI clock; scans do not own or decrement a mutable timer.
- Generic CSV readiness survives exclusive locks and resumes automatically after the same file becomes readable.
- System Burn-In is strict: STEP71 starts a 7,200-second soak, STEP72 must match `SYSTEM_ACCURACY_TEST_DATA_AVG`, and that matching capture must age for 60 seconds. Readiness is `max(step71_start + 2h, step72_start + 1m)`.
- A STEP72-only unprocessed folder is not processed. A previously persisted pass remains passed.
- Direct single-task and batch processing repeat the readiness preflight; the UI cannot bypass it.
- Not-ready results are retryable and are not persisted as processor outcomes.
- The Changeover trigger is the stable task ID `208v-breaker-8-20% Load`. That task is STEP41. STEP42 is the manual Changeover action; STEP43 is `415v-transformer`.
- Changeover is default-enabled, has its own floor event toggle and durable per-unit receipt, and is excluded from `shift_log.json`.
- Dynamic floor settings use schema v2 only when the first dynamic identity is created. Schema-v1 floors remain readable and retain the four historical IDs.
- Identity IDs are backend-generated, immutable, lowercase/path-safe, and never derived as mutable display-name keys.
- Catalog operations are add and rename only. Identity deletion and role mutation are excluded from this slice.
- Admin identities may be selected as this PC's local identity and appear on Test Ping/runtime status cards. They cannot be Main poster identities and cannot appear in end-of-shift included-station controls.
- Every PC sharing the floor must be upgraded before the first dynamic identity is created.
- OneDrive remains last-synchronized-write-wins across local replicas. The existing lock protects same-machine or true shared-filesystem read-modify-write operations, not distributed OneDrive replicas.
- Notification and floor-sync failures remain soft failures and must never block CSV processing, report writes, or the transition to STEP43.

## Current behavior and root causes

### Countdown reset while a CSV is locked

`OperatorPanel.tsx` currently decrements `remainingSeconds` once per second. Every three seconds `useTaskRunner.ts` rescans the unit folder, `applyFolderSummary` replaces task facts, and `remainingSecondsForTasks` reconstructs the time. A persisted `waiting` state receives the task's full duration, so the display falls for roughly three seconds and jumps back to the starting value.

The runner also retries only when `latestState === "detected"`. `unit_state.json` currently preserves `waiting`, so a locked CSV can stay stuck after it becomes readable.

### STEP71 incorrectly leads to a burn-in processing attempt

`system-burn-in` has detection steps `[71, 72]`, but the current frontend assigns the combined task one 7,200-second duration. STEP71 therefore ends the visible wait. The backend processor correctly needs a matching STEP72 data file, so it then returns a missing-data warning. The current UI does not gate the call on the backend's `processable` fact and there is no separate one-minute STEP72 capture phase.

### Changeover boundary

The final 208V breaker task is generated by `18 + (8 - 1) * 3 + 2 = 41`. It is not STEP42. STEP42 is a manual gap in the automation sequence. The reliable trigger is therefore the committed pass result for stable task ID `208v-breaker-8-20% Load`, while the card copy names STEP42 as the action and STEP43 as the next automated test.

### Fixed station catalog

`KNOWN_STATIONS`, `is_known_station_id`, floor validation, app-settings validation, summary filtering, shared layout creation, TypeScript defaults, and the Advanced UI all assume exactly four IDs. Replacing the four inputs with a combobox without changing this schema would cause new entries to be discarded on save. The storage and validation model must change with the UI.

## Automation readiness contract

### DTO additions

`TaskStatus` and the matching TypeScript `BackendTaskStatus` gain:

```text
process_ready: boolean
wait_phase: awaiting_csv | timing | soaking | waiting_step72 |
            capturing | waiting_unlock | ready
phase_deadline_ms: integer timestamp or null
pending_duration_seconds: non-negative integer
nominal_duration_seconds: non-negative integer
```

`processable` remains in v0.2.15 for compatibility and diagnostics, but the runner and processing preflight use `process_ready` as the authoritative gate.

The display derives one task's remaining time as:

```text
deadline_remaining = max(0, ceil((phase_deadline_ms - now_ms) / 1000))
visible_remaining  = deadline_remaining + pending_duration_seconds
```

If there is no active deadline, the phase controls the display:

- `awaiting_csv`: show the task's nominal duration.
- `waiting_step72`: show 60 seconds pending and the text “Waiting for STEP72 burn-in capture”.
- `waiting_unlock`: show zero with “Waiting for CSV to unlock”; never restart the elapsed duration.
- `ready`: show zero.

### Generic task state machine

| Filesystem fact | Phase | Ready | Deadline behavior |
|---|---|---:|---|
| Required CSV absent | `awaiting_csv` | No | None; nominal duration remains |
| Required CSV present, age below duration | `timing` | No | Creation-time anchor + duration |
| Duration elapsed, CSV unreadable | `waiting_unlock` | No | Preserve original elapsed deadline |
| Duration elapsed, CSV readable | `ready` | Yes | Preserve deadline for diagnostics |

Transformer tasks use 60 seconds. Other non-system-burn-in tasks use the existing 180 seconds.

### System Burn-In state machine

| Facts | Phase | Ready | Visible time |
|---|---|---:|---:|
| STEP71 absent | `awaiting_csv` | No | 02:01:00 |
| STEP71 present, soak incomplete, STEP72 absent | `soaking` | No | soak remainder + 01:00 |
| STEP71 soak complete, matching STEP72 absent | `waiting_step72` | No | 00:01:00 pending |
| Matching STEP72 present, later readiness boundary not reached | `soaking` or `capturing` | No | time to `max(soak, capture)` |
| Both boundaries elapsed, STEP72 locked | `waiting_unlock` | No | 00:00:00 plus lock message |
| Both boundaries elapsed, matching STEP72 readable | `ready` | Yes | 00:00:00 |

A STEP72 match requires step number 72 and filename fragment `SYSTEM_ACCURACY_TEST_DATA_AVG`. An unrelated STEP72 file does not start or satisfy the capture. STEP71 is mandatory for every new, unprocessed burn-in result.

### Processing preflight

Both `process_task` and each item in `process_tasks_with_progress` rescan and evaluate readiness immediately before calling a processor. If not ready, the command returns a `TaskProcessResult` with state `waiting`, code `2`, and an operator-readable phase message. That result is not passed to `record_processor_result` or `record_processor_results`, and workbook patch computation is not entered.

Existing `unit_state.json` entries with state `waiting` become scan-refreshable. Existing terminal pass/fail/warning and explicit acceptance semantics remain durable. A historical `system-burn-in` pass remains idempotently passed without demanding current STEP71/72 files.

### React runner and countdown

React removes the mutable one-second decrement of `remainingSeconds`. It maintains `nowMs`, updated once per second only while useful, and derives the total from the latest backend task facts. Replacing facts during a three-second scan cannot move an unchanged absolute deadline later.

Backlog selection and the current task runner use `process_ready`. A `waiting` task stays eligible for scanning and is processed automatically when the backend reports it ready. Phase-specific copy distinguishes CSV arrival, timing, STEP71 soak, STEP72 capture, and file unlock waits.

Pause, folder changes, reset, and selection changes clear the local clock context; they do not alter backend anchors.

## Changeover notification

### Trigger

The notification worker inspects committed `TaskProcessResult` values already enqueued by the Tauri commands. It attempts Changeover only when all are true:

- `result.task_id == "208v-breaker-8-20% Load"`
- `result.state == "pass"`
- `events.changeover == true`
- no durable Changeover receipt exists for the unit

Failure, warning, frontend-only Skip, another task's pass, and historical unit state observed only during scanning do not trigger the card. Batch results are safe because only committed passes are returned as committed progress/results by the existing batch path.

### Card

```text
🟡 Changeover · <identity display name>

UNIT
<serial>

ACTION
208V testing complete — shut down the PDU and change transformer taps for 415V

NEXT STEP
STEP43 · 415V Transformer Check
```

Changeover uses the Adaptive Card warning color. `NotificationEvent` maps its subject to `ACTION` and its step field to `NEXT STEP`.

### Delivery and receipt

`NotificationReceipts` gains optional `changeover: Option<NotificationReceipt>` with `#[serde(default)]`, preserving older unit-state JSON. The event key is stable per unit and boundary, for example `changeover:<serial>:208v-breaker-8-20%-load`.

The receipt is written only after Teams accepts the card. A send failure leaves no receipt so a later committed rerun can retry. Receipt write failure is surfaced in notification runtime status but does not fail automation. A saved receipt suppresses all future Changeover cards for that unit, including reruns with a different CSV.

Changeover is deliberately not appended to `shift_log.json`. That file remains Problem/Complete only so v0.2.14 shift-log readers keep working.

### Configuration compatibility

`EventToggles` gains `changeover` with a default of `true`. Existing local settings and floor files that omit the property deserialize with Changeover enabled. Advanced Settings exposes the toggle. Notification runtime status adds event kind string `changeover`.

## Dynamic identity catalog

### Floor schema

Schema v2 uses:

```json
{
  "schema_version": 2,
  "stations": [
    {
      "id": "test-station-1",
      "name": "Test Station 1",
      "role": "floor"
    },
    {
      "id": "identity-01k0example000001",
      "name": "Syed Admin",
      "role": "admin"
    }
  ]
}
```

The JSON property remains `stations` for compatibility, while UI copy uses “identity.” `StationRole` has `floor` and `admin` values. A missing role defaults to `floor`, so schema-v1 station entries migrate without changing their IDs or names.

### Version transition

- v0.2.15 accepts schema 1 and schema 2.
- A schema-v1 load supplies `role=floor` in memory and preserves `schema_version=1` during ordinary reads, renames, operator saves, password changes, and policy saves.
- Creating the first dynamic identity sets `schema_version=2` inside the floor read-modify-write lock.
- A new floor seeded without a dynamic creation may remain schema 1.
- v0.2.14 already rejects an unsupported schema number before writing; its floor overlay soft-fails to its local cache and its scoped floor save cannot load-and-rewrite schema 2. This is the required fail-closed behavior.
- All floor clients must run v0.2.15 before a dynamic identity is created. Mixed-version operation after schema 2 is unsupported even though v0.2.14 fails closed rather than deleting entries.

### Identity rules

- `id`: immutable, backend-generated for new entries, lowercase ASCII letters/digits/hyphens, maximum 64 characters, unique case-sensitively.
- Existing IDs `test-station-1`, `test-station-3`, `test-station-4`, and `pdu-lab` remain unchanged.
- `name`: trimmed, nonempty, no control characters, maximum 64 Unicode characters, unique case-insensitively.
- `role`: immutable after creation in this slice; `floor` or `admin`.
- No omission-based deletion. Saving a partial rename patch changes only named existing IDs.
- Poster and included-summary IDs must resolve to identities with role `floor`.
- Any catalog identity, including `admin`, may be selected as this PC's local `station_id`.
- Admin identities appear on Teams cards and runtime status when selected locally but do not get summary rows, Main controls, or included-station controls.

New IDs are generated under the floor lock using a time/counter-based opaque `identity-...` token, checked against the current catalog before append. Display-name edits never change the ID.

### Save request

Advanced/Connect requests add an optional operation:

```text
catalog_create:
  name: string
  role: floor | admin
  select_for_this_pc: boolean
```

Existing `stations` remains a rename patch. Under the existing floor lock the backend:

1. loads the latest floor snapshot;
2. applies explicit existing-ID renames;
3. validates and creates at most one requested identity with a generated ID;
4. changes the schema to 2 if creation occurred;
5. validates floor-only poster/included selections;
6. writes the floor atomically;
7. updates the local `station_id` to the generated ID only when `select_for_this_pc` is true;
8. returns the merged DTO containing the authoritative generated ID.

If the floor write succeeds but the local pointer write fails, the identity remains valid and unselected. A later local selection is safe. Connect to an existing floor remains adopt-only except for a separately authorized Advanced catalog save. Connect seeding may include a staged identity creation in the same locked create operation.

### Shared layout

When a new role=`floor` identity is committed, the app best-effort creates `stations/<stable-id>/` under the configured shared root. Existing directories are not renamed when display names change. Admin identities do not require a station directory in this slice. Directory creation failure is reported as a floor-sync warning and does not undo the catalog or block automation.

### Advanced UI

The current four fixed display-name fields are replaced with two controls:

1. **This PC identity** — a searchable combobox listing all floor and admin identities. Selection writes only this PC's local `station_id`.
2. **Manage identities** — an editable combobox. Selecting an existing identity loads its display name for rename. Typing a unique new name exposes explicit “Add as Floor Station” and “Add as Admin Identity” actions. An optional “Use on this PC” control sets `select_for_this_pc`.

The manager shows the immutable stable ID and role for an existing identity. It does not expose delete or role-change actions. A successful save replaces local form state with the backend response so the generated ID is immediately available in both controls.

The operator Summary Options UI receives only role=`floor` identities. Settings polling continues to apply peer changes only while the form is clean. A typed search, staged rename, or staged creation makes the form dirty and prevents peer polling from clobbering it.

Keyboard and accessibility behavior:

- labeled combobox inputs with `aria-expanded`, `aria-controls`, and `aria-activedescendant`;
- Arrow Up/Down changes the active option;
- Enter selects an identity or the explicit add action;
- Escape closes the list without discarding the typed value;
- list items use `role=option` and the list uses `role=listbox`;
- validation and save failures use the existing inline alert/result patterns.

## Storage and compatibility matrix

| Data | Storage | Compatibility behavior |
|---|---|---|
| Readiness facts | scan DTO/in memory | no new durable schema |
| Processor outcomes | `unit_state.json` | existing schema; retry waits are not persisted |
| Changeover receipt | `unit_state.json.notification_receipts.changeover` | optional/defaulted |
| Changeover toggle | local cache + `floor_settings.json.events.changeover` | missing means true |
| Identity role/catalog | local cache + `floor_settings.json.stations[]` | role defaults floor; dynamic creation writes schema 2 |
| This PC identity | local `notification_settings.json.station_id` | remains local |
| Main/included identities | floor settings | floor-role identities only |
| Shift events | `shift_log.json` | unchanged Problem/Complete schema |
| Floor station folders | `stations/<stable-id>/` | floor identities only; no renames |

## Error handling and safety boundaries

- CSV unreadability is a wait phase, not a processor failure.
- Missing STEP72 after the soak is a wait phase, not a report warning.
- Readiness preflight failure performs no workbook patch, no unit-state processor write, and no notification enqueue caused by a fabricated processor failure.
- Notification queue saturation, HTTP failure, receipt failure, and floor directory failure never alter processor results.
- Corrupt or unsupported floor JSON is never overwritten by an ordinary load/save path.
- Secrets remain redacted from Rust `Debug` output and are not shown in UI tests or screenshots.
- Dynamic ID generation occurs only inside the floor update lock and checks current IDs before append.
- Catalog patching never treats omitted identities as deletions.
- No new dependency is required.

## Non-goals

- Identity deletion, disabling, reordering, or role conversion.
- Adding Changeover counts to shift summaries.
- Retroactive Changeover messages for units whose STEP41 passed before this build.
- Treating frontend Skip as a committed Changeover boundary.
- Distributed locking or conflict resolution across OneDrive replicas.
- A file watcher or push synchronization transport.
- Changing password storage or webhook credential storage.
- Redesigning Excel mappings, workbook templates, task ordering, or breaker timing.
- Building, signing, publishing, tagging, or deploying v0.2.15.

## Automated verification requirements

### Rust automation

- STEP71 at `t0` is `soaking`, not ready, with 7,260 nominal seconds.
- STEP71 at `t0 + 2h` without STEP72 is `waiting_step72` and never enters a processor.
- STEP72 at capture age 59 seconds is not ready; at 60 seconds it is ready when soak is also complete.
- Early STEP72 waits for the later soak boundary.
- A matching locked STEP72 is `waiting_unlock`; unlocking it becomes ready without changing its timestamp anchor.
- An unrelated STEP72 filename never satisfies the burn-in capture.
- A generic locked CSV retains its original deadline across rescans.
- Direct and batch commands return retryable waits without workbook/unit-state mutation.
- Existing persisted `waiting` entries recover from scan facts.
- Existing passed burn-in remains passed.

### Frontend automation

- Fake-clock tests show that repeated three-second scans never increase remaining time for the same anchor.
- A backend `waiting` task is processed exactly once when `process_ready` becomes true.
- STEP71-only never calls a process command after two hours.
- Matching STEP72 processes only after the backend reports the one-minute capture ready.
- Each wait phase renders operator-readable copy.
- Pause/reset/folder changes do not retain a stale countdown.

### Changeover

- STEP41 failure/warning and other task passes do not send Changeover.
- First committed STEP41 pass sends exactly once.
- Receipt suppresses reruns and survives reload.
- Delivery failure stores no receipt and does not affect task processing.
- Old unit-state JSON without `changeover` loads.
- Card text and labels name STEP42 action and STEP43 next step.
- Missing `events.changeover` defaults on; disabling it suppresses delivery.
- No Changeover entry is written to `shift_log.json`.

### Dynamic identities

- Schema 1 loads all four IDs/names unchanged with floor roles and remains schema 1 without creation.
- Floor and admin creation produce safe, unique, stable IDs and schema 2.
- Rename preserves ID and role.
- Blank, control-character, overlong, duplicate names and unsafe/duplicate IDs are rejected.
- Admin IDs are rejected for poster/included arrays and accepted for local station selection.
- Dynamic local identity survives reload/poll; Connect adoption preserves the local identity.
- Concurrent scoped read-modify-write patches do not drop another identity.
- Empty/partial rename arrays never delete identities.
- Schema 2 round-trips and v0.2.14 fail-closed behavior is covered by compatibility tests/documentation.
- Combobox selection, filtering, keyboard actions, rename, both add roles, backend-generated response, and dirty-poll protection are tested.

## Manual verification before release consideration

Use a disposable unit-folder copy and a non-production notification destination first.

### Countdown and burn-in

1. Detect a normal CSV and exclusively lock it while its timer runs.
2. Observe at least three scans; the countdown must continue downward and never jump back.
3. Let the duration expire while locked; the UI must say it is waiting for unlock and must not call the processor.
4. Release the lock; the runner must process once without restarting the timer.
5. Place only the correct STEP71 burn-in CSV and verify the initial display is approximately `02:01:00`.
6. At the two-hour boundary verify no burn-in processor attempt occurs and the UI asks for STEP72.
7. Place a wrong STEP72 filename and verify it remains waiting.
8. Place the matching STEP72 capture and verify a one-minute countdown.
9. Lock the matching STEP72 through the end of the minute, then unlock it; processing must happen once.
10. Open the resulting workbook in Excel and confirm no repair prompt and preserved formatting/formulas/sheets.

### Changeover

1. Use a unit without a Changeover receipt and pass `208v-breaker-8-20% Load`.
2. Confirm exactly one yellow card with the unit serial, STEP42 action, and STEP43 next step.
3. Rerun STEP41 with another CSV and confirm no second card.
4. Disable Changeover, use a fresh unit, and confirm no card while processing continues.
5. Simulate endpoint failure on a fresh unit and confirm no receipt plus no interruption to automation.

### Dynamic identities and floor sync

1. Upgrade both test PCs to the corrective build before adding an identity.
2. On PC A, add “Syed Admin” as Admin Identity and choose “Use on this PC.” Save.
3. Confirm `floor_settings.json` has schema 2, a generated `identity-...` ID, role `admin`, and no exposed secret changes.
4. On PC B, wait for OneDrive and the 45-second clean-form poll or reopen Settings. Confirm the admin identity appears.
5. Confirm the admin identity is available for This PC identity but absent from Main and summary-included controls.
6. Add a new Floor Station on PC B. Confirm PC A receives it and `stations/<stable-id>/` exists after layout ensure.
7. Rename that station on PC A. Confirm its ID and folder remain unchanged while all labels update.
8. Keep PC B's form dirty during a peer rename and confirm its unsaved input is not clobbered; discard/reopen to receive the peer value.
9. Restart both apps and confirm each keeps its own local identity while sharing the catalog.
10. Pause OneDrive and verify automation continues with cached settings and a soft sync status.

Record source PC, setting/action, save time, OneDrive arrival time, receiving-app update time, elapsed time, and pass/fail. Do not claim multi-PC synchronization, installer behavior, updater behavior, or workbook preservation unless each was actually exercised.

## Rollout constraint

The first floor-settings write that creates a dynamic identity is the compatibility boundary. Before that action, confirm every floor and admin client using the shared `.PDU_Notifications` folder is running the corrective version. After schema 2 exists, v0.2.14 clients must be removed from settings-edit duty even though they fail closed.
